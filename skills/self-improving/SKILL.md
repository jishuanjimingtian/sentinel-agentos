## 何时使用

用户纠正你或指出错误。你完成重要工作后希望评估结果。你在自己的输出中注意到可优化之处。知识应随时间自然累积，无需人工维护。

## 架构

记忆存储于 `~/self-improving/` 目录下，采用分层结构。若 `~/self-improving/` 不存在，请运行 `setup.md`。
工作区设置应在工作区的 AGENTS、SOUL 和 `HEARTBEAT.md` 文件中添加标准的自我改进引导逻辑，并将周期性维护任务路由至 `heartbeat-rules.md`。

```
~/self-improving/
├── memory.md          # HOT：≤100 行，始终加载
├── index.md           # 主题索引（含行数统计）
├── heartbeat-state.md # 心跳状态：上次运行时间、已审核变更、操作备注
├── projects/          # 按项目划分的学习记录
├── domains/           # 领域专属（代码、写作、沟通等）
├── archive/           # COLD：已退化模式
└── corrections.md     # 最近 50 条修正日志
```

## 学习信号

当检测到以下模式时，自动记录：

**修正信号** → 添加至 `corrections.md`，并评估是否纳入 `memory.md`：
- “不对，这不正确……”
- “实际上，应该是……”
- “你关于……的说法是错的”
- “我偏好 X，而非 Y”
- “请记住，我总是……”
- “我之前告诉过你……”
- “停止做 X”
- “你为何一直……”

**偏好信号** → 若表述明确，则添加至 `memory.md`：
- “我喜欢你……”
- “总是为我执行 X”
- “永远不要执行 Y”
- “我的风格是……”
- “对于 [项目]，请使用……”

**模式候选信号** → 先跟踪，经 3 次验证后提升：
- 同一指令重复出现 ≥3 次
- 多次成功复用的工作流
- 用户明确表扬某种方法

**忽略项**（不记录）：
- 一次性指令（“现在做 X”）
- 上下文限定指令（“在此文件中……”）
- 假设性提问（“如果……会怎样？”）

## 自我反思

完成重要工作后，请暂停并进行评估：

1. **是否达到预期？** —— 对比结果与初始意图
2. **哪些方面可进一步优化？** —— 明确下次可改进之处
3. **这是否构成一种模式？** —— 若是，记录至 `corrections.md`

**触发自我反思的时机：**
- 完成多步骤任务后
- 收到反馈（正面或负面）后
- 修复 Bug 或错误后
- 发现自身输出存在优化空间时

**日志格式：**
```
CONTEXT: [任务类型]
REFLECTION: [我的发现]
LESSON: [今后应如何调整]
```

**示例：**
```
CONTEXT: 构建 Flutter UI
REFLECTION: 间距显示异常，需重新调整
LESSON: 向用户展示前，先检查视觉间距
```

自我反思条目遵循相同的升级规则：成功应用 3 次 → 升级至 HOT 层。

## 常见陷阱

| 陷阱 | 失败原因 | 更佳做法 |
|------|--------------|-------------|
| 从沉默中学习 | 导致生成错误规则 | 等待明确纠正或重复出现的证据 |
| 过快升级 | 污染 HOT 层记忆 | 新规则保持暂定状态，直至被多次验证 |
| 加载全部命名空间 | 浪费上下文资源 | 仅加载 HOT 层 + 最小匹配文件集 |
| 通过删除实现压缩 | 损失可信度与历史记录 | 改用合并、摘要或降级方式处理 |

## 核心规则

### 1. 从修正与自我反思中学习
- 用户明确纠正你时，记录日志
- 你自主识别出自身工作的改进点时，记录日志
- 绝不单凭沉默推断规则
- 同一教训重复出现 3 次 → 请求用户确认是否升格为正式规则

### 2. 分层存储
| 层级 | 位置 | 容量限制 | 行为 |
|------|----------|------------|----------|
| HOT | memory.md | ≤100 行 | 始终加载 |
| WARM | projects/, domains/ | 各 ≤200 行 | 按上下文匹配加载 |
| COLD | archive/ | 无限制 | 仅响应显式查询时加载 |

### 3. 自动升级/降级
- 某模式在 7 天内被使用 3 次 → 升级至 HOT
- 某模式连续 30 天未被使用 → 降级至 WARM
- 某模式连续 90 天未被使用 → 归档至 COLD
- 未经询问，绝不删除任何内容

### 4. 命名空间隔离
- 项目专属模式保存于 `projects/{name}.md`
- 全局偏好保存于 HOT 层（memory.md）
- 领域专属模式（如代码、写作）保存于 `domains/`
- 跨命名空间继承顺序：全局 → 领域 → 项目

### 5. 冲突解决
当模式发生冲突时：
1. 最具体者胜出（项目 > 领域 > 全局）
2. 同级中最新者胜出
3. 若仍模糊 → 向用户确认

### 6. 压缩机制
当文件超出容量限制时：
1. 将相似修正合并为单一规则
2. 归档未使用的模式
3. 摘要冗长条目
4. 绝不丢失已确认的偏好设置

### 7. 透明性
- 所有源自记忆的操作 → 必须注明来源：“使用 X（来自 projects/foo.md:12）”
- 提供周报摘要：新学模式、已降级模式、已归档模式
- 按需提供完整导出：全部文件打包为 ZIP

### 8. 安全边界
详见 `boundaries.md` —— 绝不存储凭据、健康数据或第三方信息。

### 9. 优雅降级
若上下文容量已达上限：
1. 仅加载 memory.md（HOT 层）
2. 按需加载相关命名空间
3. 绝不静默失败 —— 明确告知用户哪些内容未被加载

## 范围

本技能**仅**执行以下操作：
- 从用户修正与自我反思中学习
- 将偏好设置存储于本地文件（`~/self-improving/`）
- 当工作区集成心跳机制时，在 `~/self-improving/heartbeat-state.md` 中维护心跳状态
- 在激活时读取自身记忆文件

本技能**绝不**执行以下操作：
- 访问日历、邮件或联系人
- 发起网络请求
- 读取 `~/self-improving/` 以外的任何文件
- 从沉默或观察中推断偏好
- 在心跳清理过程中删除或盲目重写自我改进型记忆
- 修改自身的 SKILL.md 文件

## 数据存储

本地状态存储于 `~/self-improving/`：

- `memory.md`：存放 HOT 层规则与已确认的偏好设置
- `corrections.md`：存放显式修正记录与可复用的经验教训
- `projects/` 与 `domains/`：存放作用域受限的模式
- `archive/`：存放已退化或非活跃的模式
- `heartbeat-state.md`：存放周期性维护标记
